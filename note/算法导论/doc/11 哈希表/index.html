# 算法导论11 哈希表

很多应用需要一种只支持 INSERT、SEARCH、DELETE 即可的动态集合。哈希表（hashing table）是能够高效实现这种动态集合的数据结构，类似的动态集合通常被称为字典（dictionary）。一般认为哈希表中搜索的平均情况运行时间是 O(1)O(1)O(1)。

# 11.1 直接寻址

当键的全集 UUU 不大时，直接寻址（direct-address）法是一个简单实用的方法。假设 U={0,1,…,m−1}U = \{0, 1, \dots, m - 1\}U={0,1,…,m−1} 中， mmm 不太大，并且任意两个元素的键不相同。

可以使用一个数组来表示这个动态集合，该数组也被称为直接寻址表（direct-address table），记为 T[0..m−1]T[0..m - 1]T[0..m−1]。表中的每个槽（slot）对应 UUU 中的一个键。

DIRECT-ADDRESS-SEARCH(T, k)
  return T[k]
1
2
DIRECT-ADDRESS-INSERT(T, x)
  T[x.key] = x
1
2
DIRECT-ADDRESS-DELETE(T, x)
  T[x.key] = nil
1
2

每种操作都只花费 O(1)O(1)O(1) 时间。

# 11.2 哈希表

上述方法的缺点也十分明显，当 UUU 很大时，存储一个大小为 ∣U∣|U|∣U∣ 的表 TTT 是不实际的。实际存储的键构成的集合 KKK 可能远远小于 UUU，导致 TTT 中大量的空间被浪费了。

当使用直接寻址表时，键为 kkk 的元素会被存储在槽 kkk 中。而使用哈希表时，键为 kkk 的元素会被储存在槽 h(k)h(k)h(k) 中。hhh 就是哈希函数，用于计算键 kkk 对应的槽。也就是说，hhh 需要把全集 UUU 映射到哈希表 T[0..m−1]T[0..m - 1]T[0..m−1] 的槽上：

h:U→{0,1,…,m−1}h: U \rightarrow \{0, 1, \dots, m - 1\} h:U→{0,1,…,m−1}

通常，哈希表的大小 mmm 远小于全集的大小 ∣U∣|U|∣U∣。

当两个键被哈希到一个槽中时，就发生了碰撞（collision）。一般从两个方向处理这种情况，其一是直接处理碰撞，其二是避免发生碰撞。

# 使用链表解决碰撞

在这种方法中，哈希表把哈希到同一个槽中的元素以链表的形式保存起来。

CHAINED-HASH-SEARCH(T, k)
  在链表 T[h(k)] 中搜索键为 k 的元素
1
2
CHAINED-HASH-INSERT(T, x)
  把 x 插入到链表 T[h(x.key)] 的头部
1
2
CHAINED-HASH-DELETE(T, x)
  从链表 T[h(x.key)] 中删除 x
1
2

插入操作的最坏情况运行时间是 O(1)O(1)O(1)，然而这是以 x 不在哈希表中为前提的。如果插入前需要检查一遍哈希表中是否已经存有 x，那么就要先执行一次搜索。搜索的最坏情况运行时间与链表的长度成正比。如果使用双向链表，那么删除的最坏情况运行时间是 O(1)O(1)O(1)。这是因为 x 中已经存有链表前一个节点和后一个节点的信息，不需要搜索。

# 分析使用链表的哈希表

假设哈希表 TTT 有 mmm 个槽，存储了 nnn 个元素，定义 TTT 的装载因子（load factor）α=n/m\alpha = n / mα=n/m。装载因子可以理解为哈希表中链表的平均长度。

在最坏情况下，所有的 nnn 个键全部被哈希到同一个槽中，此时搜索的最坏情况运行时间是 Θ(n)\Theta(n)Θ(n)。

而平均情况运行时间取决于哈希函数 hhh 的表现。后续章节会详细讨论这个问题，现在可以假定任意键哈希到 mmm 个槽中的每个槽的概率是相等的，且与其他键的哈希结果无关，这种假设也被称为简单均匀哈希（simple uniform hashing）假设：

对于 j=0,1,…,m−1j = 0, 1, \dots, m - 1j=0,1,…,m−1，把链表 T[j]T[j]T[j] 的长度记为 njn_jnj​，可得 n=n0+n1+⋯+nm−1n = n_0 + n_1 + \dots + n_{m - 1}n=n0​+n1​+⋯+nm−1​，且有 E[nj]=α=n/mE[n_j] = \alpha = n / mE[nj​]=α=n/m。

假设计算 h(k)h(k)h(k) 只需要 O(1)O(1)O(1) 的时间，接下来需要在链表 T[h(k)]T[h(k)]T[h(k)] 上搜索。有两种结果，第一是没有搜素到键为 kkk 的元素（失败），第二是搜索到了该元素（成功）。

# 定理 11.1

在使用链表解决碰撞的哈希表中，假设使用简单均匀哈希，那么失败的搜索花费 Θ(1+α)\Theta(1 + \alpha)Θ(1+α) 的平均情况运行时间。

# 定理 11.2

在使用链表解决碰撞的哈希表中，假设使用简单均匀哈希，那么成功的搜索花费 Θ(1+α)\Theta(1 + \alpha)Θ(1+α) 的平均情况运行时间。

这意味着，如果可以做到表中元素的数量和哈希表的槽的数量成正比，即 n=O(m)n = O(m)n=O(m)，那么就可以有 α=O(m)/m=O(1)\alpha = O(m)/m = O(1)α=O(m)/m=O(1)。此时哈希表的插入、搜索、删除操作都可以在 O(1)O(1)O(1) 时间内完成。

# 11.3 哈希函数

本节将介绍三种哈希方法：除法哈希、乘法哈希和通用哈希。

# 优秀函数哈希

优秀的哈希函数满足或者渐近满足简单均匀哈希假设。不幸的是，通常没办法检查简单均匀哈希是否成立，因为键的概率分布一般是未知的。

如果能够知道键的概率分布，比如键 kkk 是满足 0⩽k<10 \leqslant k < 10⩽k<1 的实数，且满足独立均匀分布。那么：

h(k)=⌊km⌋h(k) = \lfloor k m \rfloor h(k)=⌊km⌋

就满足简单均匀哈希。

在实践中，通常使用启发式的技巧来构造表现良好的哈希函数。输入分布的定性分析往往能提供很大帮助。

# 把键转换为自然数

大多数哈希函数都假设键的全集是自然数 N={0,1,2,…}\mathbb{N} = \{0, 1, 2, \dots\}N={0,1,2,…}。当键不是自然数时，就需要将其转换为自然数。比如说，可以把字符串解释为适当进制的数字。比如字符串 "pt"，其中 p 的 ASCII 码是 112112112，t 的 ASCII 码是 116116116。"pt" 可以表示为 (112,116)(112, 116)(112,116)，如果把该序列理解为一个 128128128 进制的数字，那么 pt 相当于 112×123+116=14452112 \times 123 + 116 = 14452112×123+116=14452。

# 11.3.1 除法哈希

在使用除法（division method）构造的哈希函数中，算法利用 kkk 除以 mmm 的余数把 kkk 哈希到 mmm 个槽中：

h(k)=kmodmh(k) = k \bmod m h(k)=kmodm

使用除法哈希时，要避免使用特定的 mmm 数值。比如说，mmm 不应该是 222 的幂次。如果 m=2pm = 2^pm=2p，相当于 h(k)h(k)h(k) 只使用了 kkk 的低 ppp 比特。为了用上键中的所有比特，通常会选择一个不太接近 222 的幂次的质数作为 mmm。

# 11.3.2 乘法哈希

乘法哈希包含两个步骤。第一步，将键 kkk 乘以一个常数 AAA，其中 0<A<10 < A < 10<A<1。第二步，把 kAk AkA 的小数部分乘以 mmm，然后进行下取整操作，得到哈希值。

h(k)=⌊m(kA−⌊kA⌋)⌋h(k) = \lfloor m (kA - \lfloor kA \rfloor) \rfloor h(k)=⌊m(kA−⌊kA⌋)⌋

乘法哈希的优势在于 mmm 的选择变得不重要了，此时通常会使用 222 的幂次作为 mmm，因为这样的乘法运算更快。

# 11.3.3 通用哈希

对于固定的哈希函数来说，攻击者可以设计特殊的输入序列，使所有元素都映射到同一个槽中。通用哈希借助与键无关的随机性来实现良好的平均表现。

在通用哈希中，创建哈希表时会从特定的一系列函数中随机地选择一个作为哈希函数。注意不是每次做哈希时随机选择，而是创建哈希表时随机选择，在后续的所有操作中，选定的哈希函数是不变的。

假设 H\mathcal{H}H 是一系列能把全集 UUU 映射到 {0,1,…,m−1}\{0, 1, \dots, m - 1\}{0,1,…,m−1} 的函数。如果对于任意不同的 k∈Uk \in Uk∈U 和 l∈Ul \in Ul∈U，满足 h(k)=h(l)h(k) = h(l)h(k)=h(l) 的 h∈Hh \in \mathcal{H}h∈H 的数量不超过 ∣H∣/m|\mathcal{H}| / m∣H∣/m，那么 H\mathcal{H}H 是通用（universal）的。也就是说，随机选取哈希函数时，任意不同的 k∈Uk \in Uk∈U 和 l∈Ul \in Ul∈U 之间发生碰撞的概率不超过 1/m1 / m1/m。

假设哈希函数 hhh 是从哈希函数的通用集合中随机选出的，用于把 nnn 个键哈希到有 mmm 个槽的哈希表 TTT 中，并且使用链表来解决碰撞。如果键 kkk 不在表中，那么 kkk 对应的槽中的链表的期望长度 E[nh(k)]⩽αE[n_{h(k)}] \leqslant \alphaE[nh(k)​]⩽α。如果键 kkk 在表中，那么含 kkk 的槽中的链表的期望长度 E[nh(k)]⩽1+αE[n_{h(k)}] \leqslant 1 + \alphaE[nh(k)​]⩽1+α。

# 推论 11.4

如果使用通用哈希，使用链表解决碰撞，那么对于有 mmm 个槽的哈希表来说，处理长度为 nnn 的，含有 O(m)O(m)O(m) 个 INSERT 操作的，任意 INSERT、SEARCH、DELETE 操作的组合的期望运行时间是 Θ(n)\Theta(n)Θ(n)。

证明非常简单，因为插入元素的数量是 n=O(m)n = O(m)n=O(m)，所以有 α=n/m=O(1)\alpha = n / m = O(1)α=n/m=O(1)。因此 SEARCH 的期望运行时间是 O(1)O(1)O(1)，而 INSERT 和 DELETE 本来就只花费常数时间，所以总的运行时间是 O(n)O(n)O(n)。又由于每个操作至少是 Ω(1)\Omega(1)Ω(1)，所以最终得到 Θ(n)\Theta(n)Θ(n)。

# 设计通用哈希

首先选择一个非常大的质数 ppp，保证所有可能的键 kkk 都在区间 [0,p−1][0, p - 1][0,p−1] 中。令 Zp={0,1,…,p−1}\mathbb{Z}_p = \{0, 1, \dots, p - 1\}Zp​={0,1,…,p−1}，Zp∗={1,2,…,p−1}\mathbb{Z}_p^* = \{1, 2, \dots, p - 1\}Zp∗​={1,2,…,p−1}。可以构造：

hab(k)=((ak+b)modp)modmh_{a b}(k) = ((a k + b) \bmod p) \bmod m hab​(k)=((ak+b)modp)modm

Hpm={hab:a∈Zp∗ 并且 b∈Zp}\mathcal{H}_{p m} = \{h_{a b}: a \in \mathbb{Z}_p^* \text{ 并且 } b \in \mathbb{Z}_p\} Hpm​={hab​:a∈Zp∗​并且b∈Zp​}

当需要从 Hpm\mathcal{H}_{p m}Hpm​ 中随机选出哈希函数时，只需要随机确定 aaa 和 bbb 即可。由于一共有 p−1p - 1p−1 种 aaa，ppp 种 bbb，所以 Hpm\mathcal{H}_{p m}Hpm​ 中一共有 p(p−1)p(p - 1)p(p−1) 个哈希函数。

# 11.5 完美哈希

以上几个小节展示了哈希表能够提供良好的平均情况性能。如果键的集合是静态（static）的，那么哈希表的最坏情况性能也非常好。静态指的是一旦哈希表建立完成，键的集合就再也不会改变。

如果某种哈希在最坏情况中只需要 O(1)O(1)O(1) 次内存访问，那么就称这种哈希为完美哈希（perfect hashing）。

为了构造一种完美哈希，可以使用两级通用哈希。第一级与链表法中的哈希类似，使用通用哈希函数 hhh 把 nnn 个键哈希到 mmm 个槽中。

与链表法不同的是，每个槽中不是一个链表，而是二级哈希表（secondary hash table）SjS_jSj​。每个二级哈希表对应一个通用哈希函数 hjh_jhj​。如果能够恰当地选择 hjh_jhj​，那么二级哈希表里就可以不发生碰撞。
